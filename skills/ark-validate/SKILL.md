# /ark-validate

## 目标
验证当前工作是否满足目标，将已执行验证、建议验证、暂时无法验证的风险**严格分开**记录到 `docs/validation.md`。

## 前置建议
- 建议先执行 `/ark-test` 获得测试结果，再执行 `/ark-validate` 记录验证状态

## 适用场景
- 功能实现完成后、bug 修复后、重构后
- 阶段交付前、准备 handoff 前

## 不适用场景
- 还没有明确产出物
- 验证目标未定义
- 当前仍处在纯规划阶段

## 职责边界

**只验证和记录，不修复。**

validate 的职责是验证实现是否满足目标，并将验证状态如实记录到 `docs/validation.md`。

- **可以**：运行测试、启动服务、观察行为、执行命令、读取代码
- **不可以**：修改任何源代码文件，无论改动多小
- 发现验证失败时，**只记录失败事实**（失败项、错误信息、复现条件），不得直接修复
- 即使是一行代码的问题（如缺少 import、拼写错误），也必须交给 `/ark-debug` 处理
- 输出中必须推荐 `/ark-debug` 作为修复入口

## 输入
- `/ark-test` 的执行结果摘要（如有）
- 已修改代码或文档、当前任务目标、相关 Artifact

## 相关 Artifact
- `docs/validation.md`（主写入）

## 验证方法优先级

验证应优先使用高保真方法，从高到低：

| 优先级 | 方法 | 强度 |
|--------|------|------|
| 1 | 真实运行环境（项目实际的启动方式 + 真实配置 + 真实依赖） | 强 |
| 2 | 已有测试套件（pytest 等） | 强 |
| 3 | 自定义脚本 + 真实组件（不加 mock） | 中 |
| 4 | 自定义脚本 + mock | 弱 |

规则：
- **真实运行环境**指项目实际的启动和运行方式（从 spec.md 或 design.md 中获取）
- **不得主动创建 mock 验证脚本**——mock 测试是 `/ark-test` 的职责，不是 validate 的职责
- 如果真实环境不可用，应记入"暂时无法验证项"，**不得**降级到 mock 来凑"通过"
- 使用 mock 环境的验证结果**不得**报告为"通过"，应报告为"mock 环境通过，真实环境未验证"

## 三类验证必须分开

### 1. 已执行验证
只记录**真实执行过**的内容：跑了哪些测试、手工验证了什么、得到了什么结果。

- 不得把「理论上没问题」写成已验证
- 不得把「准备执行的验证」写成已执行

### 2. 建议验证但未执行
应该做但当前还没做的验证，必须说明为何未执行。

### 3. 暂时无法验证的风险项
因条件不足无法完成的验证，必须说明受限原因（环境依赖、数据缺失等）。
如果真实运行环境不可用（如缺少外部服务、数据库未启动等），应记入此项，不得降级为 mock 验证来凑"通过"。

## 工作流
1. 从 `/ark-test` 的输出获取已执行验证数据（如已执行）。
2. 补充手工验证、集成测试等非自动化验证结果。
3. 记录证据（命令、输出摘要、观察结果）。
4. 记录未覆盖内容。
5. 记录建议验证但未执行项（说明原因）。
6. 记录暂时无法验证项（说明受限原因）。
7. 评估验证强度：弱（主要依赖手工）/ 中（有自动化但覆盖不全）/ 强（自动化覆盖主要路径 + 手工验证边界）。
8. **如果发现验证失败**：记录失败项、错误信息、复现条件，不得修复代码，在输出中推荐 `/ark-debug`。
9. 写入 `docs/validation.md`。

## 验证要求
- 严格区分事实、建议与限制
- 若验证很弱，应直接说明而不是美化
- 验证强度评估必须有依据，不得主观拔高
- **不得修改任何源代码文件**，即使改动极小（一行 import、一个拼写错误）
- 不得修改 `docs/validation.md` 以外的核心 Artifact
- 发现验证失败时，在输出中明确推荐 `/ark-debug` 作为修复入口
- 当验证未通过但失败项被判断为既有问题时，必须同时记录三项：检查未通过的事实、失败项属于既有问题的判断、本次改动范围内是否发现新增问题。不得将此场景总结为"通过"

## 停止条件
- 当前验证状态已可读、可信、可追踪
- 未覆盖项与风险已可见
- 后续验证建议已明确

## 固定输出格式

写入 `docs/validation.md`：

### 1. 验证对象
### 2. 已执行验证（附执行命令与输出摘要）
### 3. 未覆盖内容
### 4. 建议验证但未执行（附原因）
### 5. 暂时无法验证项（附受限原因）
### 6. 风险结论
- 验证强度：弱 / 中 / 强（附依据）
- 当前可接受风险
- 当前不可接受风险
- 后续建议验证

### 7. 建议下一步

- 若验证强度不足且可补充测试 → `/ark-test` 后再 `/ark-validate`
- 若发现需要修复的问题 → `/ark-debug`
- 若验证已闭环 → `/ark-tasks` 将对应任务标记为 Done

## 备注
`/ark-validate` 的价值不在于「证明一切都好」，而在于「让真实验证状态透明可见」。
验证强度弱不是失败——诚实标注才是正确做法。
