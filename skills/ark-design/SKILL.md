# /ark-design

## 目标
形成可支撑实现的技术设计，并写入 `docs/ark/design.md`。设计的目标是降低实现阶段的主要技术歧义，而不是规定每一行代码。

## 前置建议
- 如果需求尚未规格化，建议先执行 `/ark:ark-spec` 形成书面规格
- 如果目标或范围不清晰，建议先执行 `/ark:ark-intake` 澄清任务
- 如果 `/ark:ark-analyze` 已预填充 design 内容，应审查确认现有内容，在此基础上补充完善，而非从零开始

## 适用场景
- 实现会跨多个模块
- 架构或接口选择很重要
- 重构需要明确不变边界
- 需要显式记录权衡取舍

## 不适用场景
- 改动极小，设计文档属于明显过度
- 需求本身还不清楚（应先执行 `/ark:ark-spec`）

## 输入
- `docs/ark/spec.md`（如存在）、当前代码库、技术约束、已知风险

## 输出
- 问题陈述、高层方案、模块结构、数据流/调用流、接口边界、风险与权衡、备选方案、暂不展开的细节

## 相关 Artifact
- `docs/ark/design.md`
- 可引用 `docs/ark/spec.md` 与 `docs/ark/decisions.md`

## 工作流

### 步骤一：输入源检测

执行后续步骤前，先判断是否有足够的输入：

| 条件 | 动作 |
|------|------|
| 用户提供了输入（命令后有文字） | 正常执行后续步骤 |
| 无输入，`docs/ark/spec.md` 有实质性内容且足够稳定 | 基于 spec 进入设计流程 |
| 无输入，`docs/ark/spec.md` 有内容但存在明显未决边界 | 停止，建议先 `/ark:ark-spec` 确认边界 |
| 无输入，`docs/ark/spec.md` 无实质性内容 | 停止，建议 `/ark:ark-spec` |

**"实质性内容"判定**（满足任一即可）：
- 已包含明确的能力边界或范围描述
- 已包含 analyze 预填充且不只是模板占位注释
- 已有 2 个以上非模板章节被填充

**"足够稳定"判定**：spec.md 无明显的开放问题或未决边界（不要求完美，但核心范围应已明确）。

### 后续步骤

2. 明确技术问题本身（不要跳过直接写方案）。
3. 在合适抽象层描述方案——足够具体但不过度规定实现。
4. 定义模块边界与职责，确保职责不过度重叠。
5. 记录关键流转路径与接口（函数签名、API、事件等）。
6. 写明风险与权衡——为什么当前方案优于备选？
7. 说明备选方案及未采用原因。
8. **显式标注哪些细节故意留给实现阶段**，写入「暂不展开的实现细节」节，避免过度设计。

## 验证要求
- 设计应能支撑实现，但不必规定到每一行代码
- 模块职责应清晰、不过度重叠
- 每个关键选择都应有对应理由，而不只是陈述方案
- 无用户输入时，不得在 spec.md 无实质性内容或存在明显未决边界的情况下自行生成设计
- 不得修改 `docs/ark/design.md` 以外的核心 Artifact（如 spec.md、plan.md 等）；若规格也需要更新，应建议用户显式执行 `/ark:ark-spec`

## 停止条件
- 实现阶段所需的主要技术歧义已降低到可接受程度
- 判断依据：拿着 design，实现者知道从哪里开始、边界在哪里

## 固定输出格式

写入 `docs/ark/design.md`，结构参考模板：
- 问题陈述 → 方案概述 → 模块/组件结构 → 数据流/调用流 → 接口边界 → 风险与权衡 → 备选方案 → 暂不展开的实现细节

## 建议下一步

- 若需要拆解实施步骤 → `/ark:ark-plan`
- 若计划非常简单（1-2 步）且无需状态追踪 → `/ark:ark-implement`
- 否则 → `/ark:ark-plan` → `/ark:ark-tasks` → `/ark:ark-implement`

## 备注
对于重构，必须在「接口边界」节显式标注哪些外部行为不能改变。这是重构任务中最重要的约束。
