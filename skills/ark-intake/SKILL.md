# /ark-intake

## 目标
在执行前澄清任务，识别目标、范围、约束、风险和合适的流程路径。输出应足以直接驱动 spec、plan 或 implement 的开始。

## 适用场景
- 收到一个新请求时
- 任务描述存在歧义
- 范围不清晰
- 任务可能需要规划或落 Artifact

## 不适用场景
- 任务已经非常清晰且正在执行中
- 只是一个很窄的后续动作

## 输入
- 用户请求、当前仓库状态、已有文档与代码、当前任务上下文

## 相关 Artifact
- 可引用 `docs/spec.md`、`docs/plan.md`、`docs/handoff.md` 作为上下文
- 不直接写入任何 Artifact（输出作为下游 Skill 的输入）
- **状态落点约束**：当任务规模 ≥ Medium，且输出包含关键范围判断、约束识别或阻塞性歧义时，必须推动核心结论落入 `spec` / `plan` / `handoff` 中之一，不得仅停留在对话中

## 工作流

### 步骤一：输入模糊度分级

阅读请求后，在行动前评估输入清晰度：

| 模糊度 | 判断依据 | 策略 |
|--------|---------|------|
| **Low** | 目标明确、范围可界定、技术名词具体 | 正常流程，快速完成 |
| **Medium** | 有方向但范围模糊、缺少关键约束、技术细节不清 | 引导式提问缩小范围 |
| **High** | 只有情绪/抱怨、无法描述想要什么、表述为否定句（"不好用"） | 示例驱动启发，帮助用户从具体场景出发表达需求 |

分级信号：
- **Low**：请求中包含至少一个可操作的动词 + 一个具体对象（"给用户表加邮箱字段"）
- **Medium**：有对象但动词模糊（"优化用户管理"），或有动词但对象宽泛（"改进性能"）
- **High**：无明确动词或对象，表述为感受/否定/泛化（"这个太慢了"、"不太好用"、"想重做"）

### 步骤二：上下文采集

如有必要，查看相关代码和文档（尤其是 handoff、plan、spec）。

### 步骤三：核心分析

1. 用更具体的方式重述目标——确保理解与用户请求一致。
2. 识别已知约束（技术、时间、依赖、环境）。
3. 识别已知假设，明确标注哪些是假设、哪些是事实。
4. 判断任务规模：small / medium / large（依据 `${CLAUDE_PLUGIN_ROOT}/rules/task-sizing-summary.md`）。
5. 根据规模映射出推荐的流程路径。若任务涉及理解陌生代码库或接手已有项目，推荐路径应以 `/ark-analyze` 为起点。

### 步骤四：按模糊度分流处理

**Low fuzziness** — 直接输出分析结果，无需额外澄清。

**Medium fuzziness** — 使用引导式提问缩小歧义。提问规则：
- 每个问题必须是封闭式或选择式（给出 2-3 个选项），不得使用开放式问题
- 每个问题必须附带简短的选项说明，帮助用户理解选项含义
- 问题总数不超过 3 个
- 禁止出现以下无效提问模式：
  - 「你想怎么做？」——过于开放
  - 「还有什么要补充的？」——泛问
  - 「你的需求是什么？」——等同重问

提问修正示例：
- 无效：「你想怎么优化？」→ 有效：「优化的重点方向是？（A）数据库查询耗时（B）接口响应延迟（C）内存占用过高」
- 无效：「功能范围是什么？」→ 有效：「本次需要覆盖的是？（A）仅用户注册流程（B）注册+登录（C）完整用户管理」

**High fuzziness** — 使用示例驱动启发，帮助用户从具体场景表达需求：
- 提供 2-3 个常见场景的规范表达作为参照，让用户选择最接近的或在其基础上修改
- 每个场景示例包含：用户可能的原话 → 对应的专业表达 → 涉及的典型范围
- 若场景示例仍无法触发用户表达，询问一个最具体的切入问题（如「哪个页面 / 哪个操作 / 什么时候出现的？」）
- 问题总数不超过 3 个

示例驱动模板（按需选用，不必全部展示）：

| 用户原话 | 专业表达 | 典型范围 |
|---------|---------|---------|
| "太慢了" | 优化 X 操作的响应时间，目标从 Y 秒降至 Z 秒以内 | 性能诊断 + 定点优化 |
| "不好用" | 改进 X 流程的交互体验，解决 Y 具体痛点 | 用户流程分析 + UI/UX 改进 |
| "想重做" | 重构 X 模块，保持 Y 外部行为不变，改善 Z 内部结构 | 结构分析 + 重构 |
| "加个功能" | 在 X 模块新增 Y 能力，支持 Z 场景 | 需求澄清 + 实现 |
| "出 bug 了" | X 操作触发了 Y 异常，复现路径为 Z | 症状确认 + 根因定位 |

## 验证要求
- 对任务的理解必须与用户请求一致（不得自行添加目标）
- 假设必须显式标注，不得当作事实处理
- 规模判断必须给出依据
- 模糊度判断必须给出依据（基于分级信号，而非主观感受）
- 提问必须符合当前模糊度级别的策略要求（Medium 必须用选择式，High 必须先给示例）

## 停止条件
- 任务已被澄清到可以继续推进（下游 Skill 可以直接使用输出）
- 或已明确指出阻塞性歧义，并提出了具体澄清问题
- 或 High fuzziness 经两轮启发后仍无法明确方向，如实告知用户并建议提供更具体的信息后再来

## 固定输出格式

### 1. 输入模糊度评估
**模糊度：** low / medium / high
**依据：** 列出判断依据（引用分级信号）
**处理策略：** 正常流程 / 引导式提问 / 示例驱动启发

### 2. 任务理解
用 1-3 句话重述任务目标（不加任何扩展）

### 3. 规模判断
**规模：** small / medium / large
**依据：** 列出判断依据（影响范围、风险、跨会话性等）
**推荐流程：** `intake → plan → ...`

### 5. 假设（明确标注）

### 7. 澄清问题（如有）
列出按模糊度策略提出的引导式提问或示例驱动提问。Low fuzziness 时此节可省略。

## 备注
在目标或范围不清时，不要直接进入大范围实现。提出聚焦型问题比猜测更有价值。

模糊输入不是用户的错误——用户可能不知道如何表达需求。intake 的职责是提供合适的脚手架，帮助用户把模糊意图转化为可操作的技术请求，而不是在模糊输入面前停摆或强行猜测执行。
