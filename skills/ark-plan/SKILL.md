# /ark-plan

## 目标
将目标转化为可执行、可验证的计划，并记录到 `docs/ark/plan.md`。计划的核心是「可恢复」：中断后可以从 plan 回到正确轨道。

## 适用场景
- 任务属于中型或大型
- 需要按步骤推进执行
- 多步骤任务需要可恢复性
- 需要提前定义验证策略

## 不适用场景
- 任务很小，可以安全地一步完成
- 当前需求仍过于模糊，无法有效规划（应先执行 `/ark:ark-intake` 或 `/ark:ark-spec`）

## 输入
- 已澄清的任务目标、`docs/ark/spec.md`（如存在）、`docs/ark/design.md`（如存在）、当前仓库状态、已知约束

## 输出
- 目标、范围、非范围、约束、执行步骤、里程碑、风险、阻塞项、验证策略、当前状态

## 相关 Artifact
- `docs/ark/plan.md`
- 可选引用 `docs/ark/spec.md`、`docs/ark/design.md`、`docs/ark/tasks.md`

## 工作流
1. 阅读现有 Artifact 和仓库上下文，了解已有基础。
2. 明确具体目标与本次范围，显式写出「非范围」防止 scope 漂移。
3. 识别假设、前提和约束。
4. 将任务拆为有顺序的可执行步骤——每步应产生可观察的进展。
5. 在必要时设置里程碑（检查点，而非纯粹的时间节点）。
6. 记录关键风险（发生概率 + 影响 + 应对思路）。
7. 定义验证策略：完成后如何确认目标已达成？验证方式必须可执行，不能是「看看效果」。
8. 写入 `docs/ark/plan.md`，标注状态为 not started。
9. 输出建议下一步：若计划步骤 > 2 或需要状态追踪 → 建议 `/ark:ark-tasks`；仅当计划 ≤ 2 步且无需状态追踪 → 可建议 `/ark:ark-implement`。

## 验证要求
- 每一步都应可执行，有明确的完成信号
- 验证策略必须可执行，不能是模糊描述
- 计划中不应夹带隐性 scope 扩张
- 阻塞项应记录解除条件
- 不得修改 `docs/ark/plan.md` 以外的核心 Artifact

## 停止条件
- `docs/ark/plan.md` 已足以指导后续实现（任意步骤的起点是清晰的）
- 或已明确指出妨碍规划的关键歧义

## 固定输出格式

写入 `docs/ark/plan.md`，结构参考模板：
- 目标 → 范围 → 非范围 → 前提条件 → 约束 → 执行步骤 → 里程碑（checklist）→ 风险 → 阻塞项 → 验证策略 → 当前状态 → 变更记录

## 备注
优先编写最小、可测、可恢复的计划，而不是过度细分的计划。步骤过细的计划在执行时容易过期失真。
