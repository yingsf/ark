# /ark-test

## 目标
为当前代码修改创建、补充或整理测试，输出可直接引用的执行结果，供 `/ark:ark-validate` 使用。

## 前置建议
- 如果实现目标或改动范围不清晰，建议先完成 `/ark:ark-implement` 或 `/ark:ark-debug` 再来测试

## 适用场景
- 新功能需要测试
- bug 修复需要回归覆盖
- 当前行为需要验证
- 重构需要更强的安全网

## 不适用场景
- 当前还没有可测试行为（实现尚未完成）
- 请求纯粹是文档更新

## 输入
- 已变更代码、预期行为、现有测试、失败场景或边界条件

## 输出
- 新增或更新的测试文件
- 测试执行命令与结果摘要（通过 / 失败 / 跳过数量）
- 对未覆盖区域的说明
- 可直接写入 `docs/ark/validation.md` 的验证记录草稿

## 相关 Artifact
- 可为 `docs/ark/validation.md` 提供输入（测试结果摘要）

## 工作流
1. 识别最重要的目标行为（从 spec 或 implement 的目标出发）。
2. 优先编写面向行为的测试，而非实现细节测试。
3. 对 bug 修复尽量补充回归测试，确保缺陷不再复现。
4. 在边界条件显著影响正确性时补充相关覆盖。
5. 保持测试可读、可维护，测试名称应让意图一眼可见。
6. 执行测试，记录执行命令与结果摘要：
   ```
   uv run pytest tests/ -v
   ```
7. 输出结果摘要，供 `/ark:ark-validate` 直接引用。

## 验证要求
- 测试应能在没有修复时为正确原因失败（可以证伪）
- 测试应描述可观察行为，不应测试私有实现细节
- 若无法执行测试，应明确说明原因

## 停止条件
- 当前范围内的重要行为已有合理测试覆盖
- 测试执行结果已输出（可被 `/ark:ark-validate` 引用）

## 固定输出格式

### 1. 测试目标
### 2. 新增 / 更新的测试（文件路径 + 覆盖的行为）
### 3. 执行命令与结果摘要
```
命令：uv run pytest ...
结果：X passed, Y failed, Z skipped
```
### 4. 未覆盖内容（说明原因）
### 5. 建议下一步（通常：`/ark:ark-validate` 记录验证状态）

## 备注
测试的价值不在于「覆盖率数字」，而在于「当行为出错时能第一时间发现」。
