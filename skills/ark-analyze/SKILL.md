# /ark-analyze

## 目标
扫描并理解一个已有代码库，输出项目架构认知，并可基于分析结果预填充 `docs/ark/spec.md` 和 `docs/ark/design.md`。

## 适用场景
- 接手已有项目，需要快速建立全局认知
- 对陌生代码库需要理解架构和模块关系
- 在现有项目上启用 ark 工作流前，需要先理解代码
- 重构前需要重新梳理模块职责与依赖关系

## 不适用场景
- 项目从零开始，还没有代码可分析
- 已经非常熟悉代码库，不需要重新分析
- 只需要理解某个局部函数，不需要全局视图

## 前置建议
- 建议先执行 `/ark:ark-init`（已有项目模式）植入工作流文件，再执行 analyze

## 输入
- 项目目录结构、已有代码文件、pyproject.toml / setup.py / requirements.txt、配置文件

## 输出
- 项目概览（技术栈、布局、入口）
- 模块地图（各模块职责与依赖关系）
- 核心数据流与调用链
- 外部接口清单
- 可选：预填充 `docs/ark/spec.md` 和 `docs/ark/design.md`

## 相关 Artifact
- 可预填充：`docs/ark/spec.md`、`docs/ark/design.md`
- 可引用：`docs/ark/plan.md`、`docs/ark/tasks.md`（如已存在）

## 核心原则：增量锚定分析

本 skill 采用**增量锚定分析**（Incremental Grounded Analysis）策略，将分析拆分为多个阶段，每阶段将发现的事实写入**分析工作文档**（`docs/ark/.ark-analysis.md`），下阶段从文件读取而非依赖上下文记忆。

这样做的目的是：
- 避免因上下文压力导致分析结果遗漏或虚构
- 每个阶段只关注有限信息，减少上下文压力
- 发现的事实被持久化，不会因上下文压缩而丢失
- 出错时可追溯到具体阶段的事实记录

## 分析工作文档

文件路径：`docs/ark/.ark-analysis.md`

每个分析阶段必须：
1. **先读**工作文档中已有内容（不从上下文记忆取）
2. **执行**当前阶段的分析任务
3. **写入**本阶段发现的事实到工作文档

工作文档包含以下固定章节（按阶段逐步填充）：

```
# Analysis Work Document

## 结构事实
<!-- 阶段一写入：目录树、布局类型、包名 -->

## 技术栈事实
<!-- 阶段一写入：框架、依赖、工具链、Python 版本 -->

## 入口事实
<!-- 阶段一写入：入口文件、启动流程 -->

## 模块事实
<!-- 阶段二写入：各模块职责、导入关系、分层结构 -->

## 数据流事实
<!-- 阶段二写入：请求路径、外部接口、外部系统交互 -->

## 不确定项
<!-- 阶段三写入：无法确认的内容，标注具体原因 -->
```

## 工作流

### 阶段一：结构发现

**目标**：建立项目骨架的事实基础。本阶段只做目录级别和配置文件的扫描，不深入读取业务代码。

1. **扫描目录树**
   - 列出项目目录树（排除 `.venv`、`venv`、`__pycache__`、`.git`、`node_modules`、`.mypy_cache`、`.pytest_cache`、`.ruff_cache`、`dist`、`build`、`*.egg-info`、`.tox`、`.nox`、`htmlcov`、`.coverage`、`.DS_Store`）
   - **默认记录结构摘要**：顶层目录、关键子目录、入口文件、配置文件、主要模块。不展开每个文件
   - **仅在以下条件时附录完整目录树**：项目文件总数 < 50，或用户明确要求

2. **识别项目骨架**
   - 识别项目布局类型（src layout / flat / 混合）
   - 识别包名（从 `pyproject.toml` 的 `name` 字段 > 根目录下的包目录名 > 当前目录名）
   - 识别顶层模块列表

3. **读取项目配置**
   - 读取 `pyproject.toml` 或 `setup.py` 或 `requirements.txt`
   - 记录 Python 版本要求、框架、关键依赖、包管理工具

4. **定位入口文件**
   - 查找入口文件（main.py、app.py、manage.py、cli.py、__main__.py 等）
   - 读取入口文件，记录启动流程概要

5. **写入工作文档**
   - 创建 `docs/ark/.ark-analysis.md`
   - 写入「结构事实」、「技术栈事实」、「入口事实」三个章节
   - 所有内容必须是**直接观察到的事实**（目录存在/不存在、文件内容），不做推测

### 阶段二：内容采集

**目标**：在阶段一的事实基础上，深入读取各模块内容，记录模块职责和关系。**必须先读取工作文档中的阶段一结果，再开始本阶段。**

1. **读取工作文档**
   - 完整读取 `docs/ark/.ark-analysis.md` 中阶段一已写入的内容
   - 基于阶段一的目录结构事实，确定需要深入分析的模块列表

2. **逐模块读取**
   - 按模块逐一读取关键文件（`__init__.py`、核心类定义、路由定义等）
   - 每读一个模块，立即记录：
     - 该模块的职责（基于类/函数定义和命名）
     - 该模块的 import 依赖关系
     - 该模块在分层结构中的位置
     - 每条记录标注事实类型：`[fact]`（直接观察）、`[inferred]`（合理推断）、`[uncertain]`（不确定）
   - 记录时**对照阶段一的目录结构事实**，确保描述的路径与实际结构一致

3. **追踪数据流**
   - 从入口文件出发，追踪核心请求/调用路径
   - 识别外部接口（HTTP API 端点、CLI 命令、事件处理器、定时任务等）
   - 识别与外部系统的交互（数据库、缓存、消息队列等）

4. **写入工作文档**
   - 将模块事实写入「模块事实」章节
   - 将数据流事实写入「数据流事实」章节
   - 所有内容必须基于**实际读取到的文件内容**，不得凭记忆或推测补充

### 阶段三：验证与补全

**目标**：验证阶段一和阶段二的发现，识别不确定项，补全遗漏。**必须先读取工作文档中的全部已有内容。**

1. **读取工作文档**
   - 完整读取 `docs/ark/.ark-analysis.md` 中阶段一和阶段二写入的全部内容

2. **事实核验**
   - 优先核验以下类型的事实（不是一视同仁地核验所有内容）：
     - 高影响事实：对架构判断有决定性作用的结论
     - 预填充相关：将用于预填充 spec/design 的事实
     - `[inferred]` 和 `[uncertain]` 标记的事实
   - 对工作文档中记录的上述事实，验证其源文件是否确实存在且内容匹配
   - 如果发现矛盾或无法确认的内容，记入「不确定项」

3. **识别不确定项**
   - 标记所有无法从代码中确认的内容
   - 每个不确定项必须说明：不确定的是什么、为什么无法确认
   - 内部分析时使用 Artifact 标签辅助判断：
     - 关于"系统做什么、有什么能力"→ `[spec]`
     - 关于"系统怎么构建、模块关系"→ `[design]`
     - 两者都涉及 → `[spec, design]`
   - **输出时按后续动作分组**（见固定输出格式第 7 节），不得仅以标签形式输出。分组仅用于帮助用户判断后续处理方向，不表示 `/ark:ark-analyze` 会自动执行对应 Skill
   - **不得将不确定的内容写成确定事实**

4. **写入工作文档**
   - 将不确定项写入「不确定项」章节
   - 修正工作文档中发现的事实错误

### 阶段四：报告生成与预填充

**目标**：基于工作文档中的已验证事实，生成最终分析报告并预填充 Artifact。**必须先读取工作文档的全部内容。**

1. **读取工作文档**
   - 完整读取 `docs/ark/.ark-analysis.md` 的全部内容

2. **生成分析报告**
   - 基于工作文档中的事实（不基于上下文记忆），按「固定输出格式」生成报告
   - 报告中的每一项都应能在工作文档中找到对应的事实依据

3. **预填充 Artifact（可选）**
   - 预填充 `docs/ark/spec.md`：
     - 背景：项目是什么、解决什么问题
     - 目标：当前系统提供的核心能力
     - 范围：已知的功能范围
     - 验收标准：基于现有行为描述
   - 预填充 `docs/ark/design.md`：
     - 模块结构：各模块及其职责
     - 数据流/调用流：核心路径
     - 接口边界：外部接口清单
   - 预填充内容必须标注 `<!-- 由 /ark:ark-analyze 自动生成，需人工确认 -->`

4. **清理工作文档**
   - 询问用户是否保留 `docs/ark/.ark-analysis.md`
   - 若用户选择不保留，删除该文件
   - 若用户选择保留，该文件可作为后续调查的参考

### 不确定项回答响应

本节描述分析报告输出后，用户对不确定项给出回答时的处理流程。

1. **消化用户回答**
   - 将用户的回答与对应 U 编号关联
   - 判断每条回答是否足以消除不确定性

2. **生成建议更新方案**
   - 对每条回答明确其去向：
     - → `docs/ark/spec.md`（规格边界确认）
     - → `docs/ark/design.md`（设计机制确认）
     - → `docs/ark/.ark-analysis.md`（补充事实记录）
     - → 暂不更新（信息不足或 E 类）
   - 注明每条更新的具体内容摘要

3. **等待用户确认**
   - 输出建议更新方案后停止，等待用户确认
   - 用户可调整方案（修改去向、内容、跳过某些项）

4. **执行确认后的写入**
   - 仅写入 `docs/ark/.ark-analysis.md`：更新对应不确定项状态为"[已澄清]"（用户已回答，但正式 spec/design 更新仍待 `/ark:ark-spec`、`/ark:ark-design` 完成），记录用户回答摘要
   - 对于涉及 spec/design 的更新建议，明确列出：
     - 哪些 U 项需通过 `/ark:ark-spec` 处理
     - 哪些 U 项需通过 `/ark:ark-design` 处理
     - 哪些 U 项需先 `/ark:ark-spec` 再 `/ark:ark-design`

**禁止事项：**
- 不得写入 `docs/ark/spec.md` 或 `docs/ark/design.md`（无论用户是否确认）；对 spec/design 的更新建议只能通过显式执行 `/ark:ark-spec`、`/ark:ark-design` 完成
- 仅可写入 `docs/ark/.ark-analysis.md`（更新不确定项状态、记录用户回答）
- 不得将用户对不确定项的回答自动写入 memory
- 不得自动进入 `/ark:ark-spec`、`/ark:ark-design` 或其他 Skill 的职责范围

## 回写规则

### 预填充 `docs/ark/spec.md`
- 仅在 spec.md 为空或为初始模板时预填充
- 预填充内容基于工作文档中的已验证事实，不是凭空编写
- 必须标注来源

### 预填充 `docs/ark/design.md`
- 仅在 design.md 为空或为初始模板时预填充
- 基于工作文档中的已验证事实
- 必须标注来源

### 分析工作文档
- 必须在阶段一开始时创建
- 每个阶段结束前必须写入该阶段的发现
- 下个阶段开始时必须先读取已有内容
- 所有写入内容必须是可追溯到源文件的事实

## 验证要求
- 分析结果必须基于工作文档中的事实，不得凭上下文记忆填充
- 每个阶段必须先读取工作文档再开始工作
- 工作文档中记录的路径、文件名必须与实际目录树一致
- 模块职责描述应有代码依据（如导入关系、类/函数定义）
- 外部接口应从代码中识别，而非假设
- 不确定的内容应标注为「待确认」，不应写成事实
- 不确定项必须说明不确定的具体内容和原因
- 收到用户对不确定项的回答时，必须先生成建议更新方案并等待确认，不得直接写入 Artifact
- 建议更新方案必须明确区分每条回答的去向（spec / design / analysis-doc / 暂不更新）
- 用户对不确定项的回答经确认后，analyze 只能更新工作文档状态；涉及 spec/design 的更新必须通过显式 `/ark:ark-spec`、`/ark:ark-design` 完成，不得直接写入

## 停止条件
- 项目架构已清晰到可以支持后续 spec/design 审查和 plan 编写
- 或已明确指出哪些部分因代码复杂度无法快速理解，需要深入分析
- 工作文档已完整写入所有阶段的发现

## 固定输出格式

### 1. 项目概览
- 项目名称
- 项目类型（Web 服务 / CLI 工具 / 库 / 其他）
- Python 版本
- 包管理工具
- 核心框架与关键依赖

### 2. 目录结构
输出关键目录树（排除缓存、构建产物、虚拟环境等），标注各目录职责。

### 3. 模块地图

| 模块 | 职责 | 依赖 |
|------|------|------|

### 4. 入口与核心路径
- 入口文件
- 核心请求/数据流概述

### 5. 外部接口

| 类型 | 名称 | 说明 |
|------|------|------|

### 6. Artifact 预填充状态
- `docs/ark/spec.md`：已预填充 / 未预填充（原因）
- `docs/ark/design.md`：已预填充 / 未预填充（原因）

### 7. 不确定项（按后续动作分组）

为每个不确定项编号（U1、U2、...），按以下类别分组输出。若某类为空则省略该组。

**A. 待确认的规格边界**
关于系统做什么、有什么能力的不确定项。
若你准备确认这些边界，可后续显式执行 `/ark:ark-spec`。

**B. 待确认的设计机制**
关于系统怎么构建、模块关系的不确定项。
若你准备确认这些机制，可后续显式执行 `/ark:ark-design`。

**C. 同时涉及规格与设计**
同时涉及规格与设计的不确定项。
若你准备正式更新文档，通常先显式执行 `/ark:ark-spec`，再显式执行 `/ark:ark-design`。

**D. 更像真实问题或配置缺陷**
分析中发现的可确认问题。`/ark:ark-analyze` 只负责暴露问题，不自动进入 `/ark:ark-plan`、`/ark:ark-implement` 或 `/ark:ark-debug`。若你准备处理，请显式选择对应 Skill。

**E. 暂不阻塞当前任务**
当前不影响理解和使用，可暂时保留，不必立即处理。

### 8. 批量处理建议与回答响应

- 你可以一次性回答多个 U 编号（如"U1: ... U2: ... U5: ..."），无需逐条处理
- 回答后，Claude 按以下格式输出建议更新方案，等待你确认后再执行：

| U 编号 | 你的回答 | 建议去向 | 更新内容摘要 |
|--------|---------|---------|-------------|
| U1 | ... | docs/ark/spec.md | ... |
| U2 | ... | docs/ark/design.md | ... |
| U5 | ... | 暂不更新 | ... |

- `/ark:ark-analyze` 不自动执行任何后续 Skill

> **C 类处理指引（实现参考，不输出给用户）：**
> 当不确定项同时涉及 spec 和 design 时，通常先用 `/ark:ark-spec` 确认系统边界、能力承诺与范围，再用 `/ark:ark-design` 解释当前结构与实现机制。
> 这样可避免将"预留扩展位"误写成"已成立能力"。
>
> 示例：不确定项"core/db/vector/ 目录名暗示向量数据库，但当前仅有 ES 相关代码"
> - `/ark:ark-spec` 先确认：该目录是为未来多向量数据库扩展预留，当前不构成正式能力承诺
> - `/ark:ark-design` 再解释：目录命名原因、当前 ES 结构现状、未来扩展方向

### 9. 建议下一步

- 若要确认系统边界、能力范围、框架约定：可显式执行 `/ark:ark-spec`
- 若要确认结构机制、模块关系、实现解释：可显式执行 `/ark:ark-design`
- 若准备推进开发，但需求、范围或成功标准仍未清晰：先显式执行 `/ark:ark-intake`
- 若目标已经明确，且需要拆分阶段与实施步骤：可显式执行 `/ark:ark-plan`

## 备注
`/ark:ark-analyze` 的目标不是产出完美的文档，而是「让陌生代码库在短时间内变得可理解」。
分析结果应作为后续 spec、design、plan 的输入，而非最终产出。
