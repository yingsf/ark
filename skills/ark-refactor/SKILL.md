# /ark-refactor

## 目标
在保持外部行为稳定的前提下改善代码结构。

## 前置建议
- 如果 Artifact 文档可能已过期，建议先执行 `/ark-sync` 同步状态

## 适用场景
- 代码存在明显可维护性或清晰度问题
- 重复与复杂度已造成实际负担
- 希望优化结构而不是新增功能

## 不适用场景
- 当前任务本质上是加新功能
- 不变行为边界不清楚
- 当前核心问题是 bug，而不是结构问题

## 输入
- 当前代码、预期不变行为、相关测试、相关 design 或 plan Artifact
- 外部检查结果（可选）：如 SonarLint、mypy、ruff 等工具输出

## 输出
- 改善后的结构、明确列出的不变行为、建议或补充的回归覆盖、剩余风险说明

## 相关 Artifact
- 可引用 `docs/design.md`
- 应为 `docs/validation.md` 提供输入

## 工作流
1. 先定义哪些行为不能变。
2. 识别需要优化的结构性问题。
3. 采用渐进式重构。
4. 避免把新功能混入同一次重构。
5. 在需要时加强回归安全网。

## 验证要求
- 不变行为必须显式化
- 应在可行范围内检查行为稳定性
- 若存在混合 scope，应主动指出

## 停止条件
- 结构目标得到改善，且行为变化风险处于可接受范围

## 备注
明确说明本次重构的非功能目标，例如：清晰度、模块化、可测试性、去重。
