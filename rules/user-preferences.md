# User Preferences

本文件记录 ark 驱动工作流中的个人稳定偏好，可随着使用逐步调整。

## 默认语言与运行时

- 首选语言：Python
- 默认 Python 版本：3.12
- 可选 Python 版本：范围从 `3.10` 到 `3.14`
- 新项目最低可接受版本：3.12

## 依赖与环境管理

- 包管理工具：uv（固定）
- 虚拟环境目录：`.venv/`

## 项目布局偏好

- 默认使用 `src/` layout
- 默认在项目根目录使用 `tests/`
- 默认使用 `docs/ark/` 保存任务状态 Artifact
- docs/ark/ Artifact 自动创建，不需要用户确认

## 工具链默认值

- 测试框架：pytest
- 代码风格/lint：ruff（lint + format）
- 静态类型检查：pyright（typeCheckingMode: standard）
- 人工检查：推荐使用 PyCharm 的 SonarQube For IDE 插件进行最终的人工代码检查

## 代码风格偏好

- 优先清晰、显式的代码，而不是聪明但晦涩的写法
- 倾向小函数、单一职责
- 倾向为公共接口补充类型标注
- 当结构化数据需要显式约束时，优先考虑 dataclass 或 pydantic model
- 当复杂度上升时，优先依赖注入而不是隐式全局状态

## 注释与 docstring 偏好

- 希望生成的代码包含清晰、充分但克制的中文注释
- 公共类、公共函数、关键方法应补充 docstring
- 复杂逻辑、边界条件、降级策略、设计原因应补充必要注释
- 注释应解释"为什么 / 约束 / 风险"，不要重复代码表面含义
- 不接受"又臭又长"的 docstring 和注释

## 文档偏好

- README 保持简洁但可操作
- 项目文档应反映真实代码状态
- 避免"好看但无执行价值"的装饰性文档

## 工作流偏好

- 中任务默认生成 `plan.md` 和 `validation.md`
- 大任务默认生成全部 7 个 Artifact
- 对需求不清的请求，优先提出聚焦型澄清问题
- 对重构任务，应显式写明不变行为约束

## 评审偏好

- 希望代码能通过架构师级别审查
- 评审优先关注正确性、失败行为、接口清晰度、可维护性、可测试性，再看安全性
- 安全和并发重要，但不应在所有场景压倒一切
- 区分确认的问题与低置信度担忧，避免风格噪音

## 测试偏好

- 优先面向行为的测试
- 在可行时覆盖边界条件
- bug 修复尽量补回归测试
