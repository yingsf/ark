# Python Backend Conventions

本文件定义 ark 在 Python 后端项目中的默认工程规范。

## 项目结构

推荐默认结构：

- `pyproject.toml`
- `src/<package_name>/`
- `tests/`
- `docs/`

新项目默认使用 `src/` layout。

## 包组织建议

典型包结构可包含：

- `api/`：接口层、传输层、HTTP 层
- `domain/`：核心领域模型与领域逻辑
- `services/`：应用服务、编排逻辑
- `repositories/`：持久化访问
- `config.py`：配置加载与设置定义
- `main.py`：应用入口

并非所有项目都必须有这些目录。只创建当前范围真正需要的部分。

## 配置管理

- 优先通过环境变量与显式 settings 对象管理配置
- 避免在各处零散读取配置
- 配置读取应尽量集中

## 类型标注

- 公共函数和方法应尽量补充类型
- 返回值类型尽量精确
- 避免不必要的 `Any`

## 测试规范

- 默认使用 pytest
- 测试按行为和模块边界组织
- 纯逻辑优先单元测试
- 跨边界的重要行为应有集成测试
- 确认 bug 后，尽量补回归测试

## 异常处理

- 抛出有意义的异常
- 避免静默失败
- 必要时在边界层将底层异常转换为更合适的领域异常

## 日志规范

- 优先结构化、带上下文的日志
- 避免正常流程中过量 debug 噪音
- 不记录敏感信息

## 代码质量原则

- 优先简单、可组合的模块
- 避免过早抽象
- 当重复或复杂度已造成实质性负担时再重构
- 接口保持小而明确

## 可维护性与结构复杂度

### 控制函数复杂度
- 单个函数应优先保持单一职责
- 当分支、循环、异常处理和组装逻辑叠加导致理解成本明显上升时，应优先拆分为辅助函数
- 避免把输入解析、业务判断、错误处理、输出组装长期堆在同一函数中

### 尊重封装边界
- 优先通过公开方法或明确提供的访问接口协作
- 避免在类外直接访问以 `_` 开头的内部成员，除非这是明确约定的内部机制且已在设计层说明
- 若确需访问内部状态，应优先考虑补公开接口或重构协作方式

### 对待重复代码的原则
- 避免复制粘贴式的大段重复实现（15 行以上且结构高度相似）
- 仅当重复已形成实质维护负担时再抽象，不为消除轻微重复而引入更复杂的抽象层
- 若暂不抽取，应说明理由（如避免过早抽象）
- 对中大型、分批实施的任务，重复代码应结合任务整体判断；若当前批次与前序批次形成跨文件重复，应在后续收口批次中统一处理，不得长期保留无主的大段重复实现

## 文档与注释规范

- 公共类、公共函数、关键方法应具备清晰 docstring
- 复杂逻辑、非直观边界、降级策略、性能/并发关键点应补充必要中文注释
- 注释优先解释"为什么这样做""约束是什么""风险在哪里"
- 类级 docstring 可以相对完整，私有 helper 保持简洁
- 避免显而易见代码的重复性注释
- 避免过长、装饰性、无信息密度的 docstring 和块注释

## 默认工具链

- `pytest`：测试
- `ruff`：lint + format（替代 flake8、isort、black）
- `pyright`：静态类型检查（typeCheckingMode: standard）
- 提交前应运行 `uv run ruff format . && uv run ruff check . --fix && uv run pyright`

## Review 关注顺序

评审 Python 后端代码时，优先关注：

1. 正确性
2. 失败行为
3. 接口清晰度
4. 可维护性
5. 可测试性
6. 运行安全性

安全、并发、性能在当前改动明显涉及这些维度时提高权重，不默认对所有改动做军规式安全审计。
